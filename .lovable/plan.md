## الهدف
رفع Performance للموبايل من 71 إلى +90/قريب من 100، مع الحفاظ على Tajawal وشكل الصفحة، والديسكتوب 100.

## ما أكده التقرير الحالي
- Mobile Performance = 71.
- FCP = 4.2s و LCP = 4.4s؛ المشكلة الأساسية تحميل/عرض أول شاشة وليس TBT، لأن Total Blocking Time = 0ms.
- أكبر نزيف ظاهر: JavaScript غير مستخدم في أول تحميل:
  - `index-*.js` حوالي 361KB.
  - `charts-*.js` حوالي 104KB ما زال يُحمّل على الصفحة الرئيسية رغم أنه خاص بالإدارة/الرسوم.
- LCP هو اللوجو `/logo.webp` داخل الهيدر، والتقرير يطلب تحسين image delivery/render-blocking.
- cache TTL في النسخة المنشورة ما زال 7 أيام لملفات assets وليس immutable طويل.

## الخطة التنفيذية
1. **إخراج charts من الصفحة الرئيسية نهائيًا**
   - إعادة هيكلة صفحات الإدارة التي تستخدم Recharts بحيث تحميل `DashboardCharts` و`AnalyticsCharts` يتم فقط عند فتح صفحات الإدارة.
   - إن لزم، تحويل مكونات الرسوم نفسها إلى lazy dynamic imports داخل الصفحة بدل static import.
   - الهدف: ألا يظهر `charts-*.js` في Network الخاص بالهوم.

2. **تقليل حزمة `index-*.js` التي تصل للموبايل**
   - مراجعة الاستيرادات التي تدخل في root/home فقط: الهيدر، الـauth/cart hooks، LanguageProvider، MaintenanceGate، وHero.
   - تأجيل أي منطق غير ضروري للـfirst paint: حساب المفضلة، عداد السلة، حالة المستخدم، dropdowns، mobile menu details.
   - جعل الهيدر في أول رسم static قدر الإمكان، ثم تفعيل العدادات وحالة الحساب بعد أول تفاعل/idle.

3. **حل LCP الخاص باللوجو**
   - تحسين أبعاد وجودة `/logo.webp` أو إنشاء نسخة أصغر مخصصة للهيدر لو الملف الحالي أكبر من المطلوب.
   - إبقاء preload/fetchPriority لكن بدون تحميل صورة أكبر من مقاس العرض الحقيقي.
   - التأكد أن اللوجو لا ينتظر CSS/JS لكي يظهر.

4. **تقليل render-blocking CSS للموبايل**
   - مراجعة CSS العام والـanimations المستخدمة في first viewport.
   - تعطيل/تأجيل animations والـblur/backdrop الثقيلة على الموبايل في الهيرو والهيدر.
   - الحفاظ على Tajawal self-hosted، لكن تحميل أقل أوزان ممكنة في أول شاشة.

5. **تثبيت caching في النشر**
   - تحديث headers لتطبيق Cache-Control طويل على `/assets/*` وليس `/assets/*` فقط إن كان مسار الاستضافة يحتاج pattern مختلف.
   - الهدف: اختفاء ملاحظة cache TTL أو تقليلها جدًا في PageSpeed.

6. **التحقق بعد التنفيذ**
   - تشغيل فحص محلي/Playwright للشبكة للتأكد أن الصفحة الرئيسية لا تطلب `charts` ولا API مبكرًا.
   - فحص حجم ملفات build الناتجة عن Vite لمعرفة هل `index` قل فعليًا.
   - إعادة فحص الصفحة المنشورة بعد Publish لأن PageSpeed يقيس `saba-design.com` وليس preview/local.

## ملاحظة مهمة
لا أقدر أضمن رقم 100 حرفيًا لأن PageSpeed mobile يتأثر بسرعة الخادم والشبكة ولحظة القياس، لكن هذه الخطة تستهدف الأسباب المؤكدة في التقرير الحالي: LCP/FCP وحجم JS غير المستخدم، وهي الطريق الواقعي للوصول إلى +90 وربما 100.
